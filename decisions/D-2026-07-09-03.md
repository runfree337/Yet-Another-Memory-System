**Decision**: a work item's STATE.md is pure state — frontmatter + a mandatory `## Tasks`
section (states `todo/in-progress/blocked/done`, label ≤ 30 words or a `→ working doc`
reference). Durable knowledge is captured at the end of each task, immediately routed to its
durable home, and never stored in STATE.md (soft guard: > 80 lines or an off-canon section).

**Why**: observed failure mode on the original host project — documentation written at task end
accumulated inside the state file, which grew into a transient duplicate of durable docs. The
closure step then became a costly migration instead of a quick control. Alternative ruled out:
free-form state rubrics — unverifiable, and the 30-word cap is what forces a working doc out of
the label.

**Invariant**: closing a work item never migrates content — by closure time the durable already
lives elsewhere and STATE.md can be deleted without loss (verifiable: `E-STATE-SIZE`,
`E-STATE-SECTION`, `E-TASK-LEN`).
