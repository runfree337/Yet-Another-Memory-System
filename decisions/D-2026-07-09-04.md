**Decision**: the runtime bridge between an agent's searches and the cartography is a
**nudge, never a rail**. A search over a covered zone always executes untouched; the framework
may only append context *next to* its raw results — pointers to the index and matching entries,
each with its `updated` date — and only on a broad sweep in a session that hasn't consulted any
channel index yet. Intercepting a search to serve cartography results *instead* is out of scope,
in every mode.

**Why**: the cartography is a derived representation of the code — served in place of it, a
stale entry becomes an authoritative lie (`FEATURE_MAP.md`: "an entry that lies is worse than
none"), undetectable precisely because the real results were suppressed. A targeted single-file
search is typically pre-edit verification, where only the file's current content counts; the
same reasoning refines the bypass metric (a targeted search is not a bypass, consulting any
channel index is a consultation). Alternative ruled out: strict interception ("serve the graph
when confident") — the system self-assesses the sufficiency of its own data, exactly the case
where it cannot detect its own staleness, and an agent routes around a blocked search tool via
the shell anyway.

**Invariant**: no YAMS hook blocks or rewrites a search or its results — the nudge is wired
`PostToolUse` (structurally unable to alter the tool's output), appends at most one note per
zone per session, and is a silent no-op everywhere else (verifiable:
`adapters/claude-code/hooks/tests/index-hooks-test.sh`).
